# HAIOS Operating Framework

*This framework governs HAIOS at the system level. It is pattern-agnostic — any specific architecture, implementation, or use case is an expression of these constraints, not the source of them. Where examples appear, they are illustrative only.*

*This framework derives from The Measure — HAIOS Foundational Theory. The timing and structure of ceremonies are grounded in the nested oscillatory logic described there. Where timing choices are empirical hypotheses rather than proven couplings, that is noted.*

## Principles

**1. Cheapest viable resolution first.**
Every decision attempts the lowest-cost path. Cost means compute, money, time, human attention. Escalation is deliberate, not default.

**2. Every execution is training data.**
Nothing runs unobserved. Every decision, outcome, and correction is captured. The system's memory is its primary asset.

**3. Confidence governs flow.**
Uncertainty is measured, not avoided. Confidence determines routing, escalation, and autonomy. Thresholds are explicit and tunable.

**4. The system teaches itself.**
Feedback loops are not optional. What is expensive today becomes cheap tomorrow. The direction of travel is always toward autonomy.

**5. New capabilities are immediately available, eventually learned.**
Any new component is usable on arrival through description alone. Repeated use graduates it into embedded knowledge. Nothing waits for a learning cycle to be useful.

**6. The measure measures itself.**
Timing, thresholds, and rhythms are hypotheses, not laws. The system generates performance data against its own operating assumptions and adapts when the data warrants it.

---

## Ways of Working

- **Measure before you optimise.** No process is improved until it's instrumented.
- **Trust the layers.** Don't bypass structure for short-term speed. Shortcuts starve the feedback loop.
- **Thresholds are hypotheses.** Boundaries are set empirically and revised regularly. None are sacred.
- **Capabilities enter declaratively.** New components are described into the system, not hardwired. The system discovers what it can do.
- **Humans are a layer, not an escape hatch.** Human decisions are logged identically to automated ones. Human involvement is structured, not ad hoc.
- **Rhythms are coupled, not arbitrary.** Ceremony timing derives from The Measure. If a rhythm has no grounding, note it and test it.

## Ceremonies

*Ceremonies follow the four-phase cycle described in The Measure: Observe → Accumulate → Consolidate → Emerge changed. Each ceremony operates at a different temporal scale, and each faster rhythm feeds the slower one above it.*

**Pulse** — *The ultradian cycle (~90-120 minutes)*
One complete observation-action-logging unit. The system's operational heartbeat. During active phases, pulses accumulate. During rest phases, the day consolidates what the pulses gathered. No ceremony is performed — the pulse *is* the work.

**Day** — *The circadian cycle (24 hours)*
The boundary between active operation and consolidation. Review system activity from the waking phase. Flag anomalies. The day's rest phase is not idle — it is when accumulated pulse data is compressed into daily signals. Awareness, not action.

**Calibration** — *The lunar cycle (~29.5 days)*
*Timing hypothesis: lunar month. To be tested against calendar month and empirical performance data.*
Analyse performance across layers over the full cycle. Where is accuracy drifting? Are thresholds appropriate? What patterns emerged from the daily signals? This is where accumulated observation becomes actionable insight.

**Evolution** — *The lunar cycle, acted upon*
Act on what Calibration revealed. Retrain, reconfigure, retire redundancies. Measure before and after. The system emerges from each evolution cycle at a different baseline than it entered.

**Expansion** — *The solar cycle (~90 days, aligned to solstice/equinox)*
*Timing hypothesis: seasonal quarter. Equinoxes and solstices as natural inflection points.*
Audit for new opportunities. Where else in the system can these principles apply? Map cost, volume, and current method. Assess whether the operating context has shifted. Prioritise.

**Orbit** — *The annual cycle (365 days)*
Strategic assessment. Is the system on a different trajectory than one orbit ago, or merely oscillating? Review the full year's evolution cycles. Recalibrate the framework itself if the data warrants it. The Measure is tested here.

## Activities

*Activities are the work performed within and between ceremonies. Each activity belongs to one or more temporal scales.*

- Logging decisions and outcomes (continuous — every pulse)
- Monitoring system confidence distributions (daily)
- Compressing pulse data into daily signals (daily — rest phase)
- Curating and cleaning accumulated data (calibration)
- Tuning thresholds and parameters (calibration)
- Describing new capabilities into the system (as needed)
- Retraining and updating learned components (evolution)
- Evaluating accuracy against baselines (evolution)
- Testing timing hypotheses against performance data (evolution)
- Mapping new application opportunities (expansion)
- Assessing operating context shifts (expansion)
- Reviewing framework assumptions (orbit)
- Documenting pattern variations and edge cases (ongoing)

---

## Assets

*Assets are what the system produces and accumulates. They compound over time — each ceremony cycle adds to them.*

- **Decision logs** — the raw history of every system action. Input, resolution path, confidence, outcome. Generated continuously at pulse level.
- **Daily signals** — compressed summaries of each day's pulse activity. Anomalies, patterns, confidence distributions. Generated during the circadian rest phase.
- **Training sets** — curated, labelled subsets of decision logs prepared for learning cycles. Produced at calibration, consumed at evolution.
- **Learned weights** — model adapters, embeddings, or any artifact that represents accumulated knowledge. Versioned. Updated at evolution.
- **Capability registry** — descriptions, schemas, and usage examples of everything the system can do. Updated declaratively as capabilities enter.
- **System configuration** — current thresholds, layer definitions, escalation rules, timing parameters. The tunable physics of the system.
- **Performance baselines** — accuracy, latency, cost metrics at each ceremony interval. The proof the system is improving.
- **Timing performance data** — how ceremony rhythms correlate with learning outcomes. The data that tests The Measure's hypotheses. Reviewed at orbit.
- **Pattern library** — documented instances of the framework applied to specific domains. The playbook for expansion.
